## 4. PREUVE RÉELLE — l'abonnement ChatGPT pilote Cowork (RÉEL)

### 4.1 Chat live avec réponse calculée

Spec d'audit : prompt demandant le calcul de **17 × 23** avec une réponse formatée sur une ligne
commençant par un jeton (`CBAUDIT`) absent du premier mot du prompt — un echo/mock ne peut donc
pas satisfaire l'assertion `^CBAUDIT`.

**Réponse verbatim du modèle (capturée par le test) :**

```
CBAUDIT 391 [tokens: 43 in / 3 out | cost: $0.0000]
```

- `391` = **17 × 23 exact** → calcul réel, pas un echo.
- Format exact respecté → instruction suivie par le modèle.
- `cost: $0.0000` → cohérent avec l'**abonnement ChatGPT OAuth** (coût marginal nul via le backend Codex,
  ≠ une API facturée au token).

Reproduit **4 fois** (specs + pilote), jetons imposés différents à chaque run, écho du prompt exclu :
`CBAUDIT 391`, `CBPILOT 391`, `CBMCP 391`. Et une **réponse en langage naturel** (prompt « explique
Cowork en une phrase, commence par CBDOC ») :

> **CBDOC** Cowork est un assistant de développement collaboratif qui aide à coder, modifier des fichiers et diagnostiquer des problèmes techniques.

→ Une réponse cohérente, pertinente et formatée selon l'instruction : c'est une **vraie sortie modèle**,
impossible à produire par un mock/fixture. **La preuve de la réponse est ce texte verbatim**, pas une capture.

### 4.2 Preuve backend indépendante — refresh du token OAuth pendant le run

Le `last_refresh` de `codex-auth.json` est passé de **`16:51:50Z`** (avant) à **`18:44:01Z`** (pendant
le run du chat réel canonique). Un appel OAuth live vers le backend Codex a donc bien eu lieu — preuve
indépendante de la GUI qu'aucun mock/cache n'a servi la réponse. Le round-trip réel a duré **2,2 min**
(le spec canonique), incompatible avec un mock instantané.

## 6. Couverture fonctionnelle — sweep e2e déterministe (DÉTERMINISTE)

> Suite Playwright complète : `cd cowork && DISPLAY=:10.0 npx playwright test` (152 tests, `workers:1`).
> Specs réels/démo auto-skippés (non concernés ici). Log : `/tmp/cowork-e2e-sweep.log`.

**Contexte essentiel — working tree en plein refactor.** Le sweep tourne contre la source
**non commitée** (43 fichiers `cowork/src` + 80 specs e2e modifiés), notamment un refactor de
la barre de menu : `TopMenuBar.tsx` est un **fichier nouveau (non-suivi)**, `MemoryBrowser.tsx`
est **supprimé**, `App.tsx` plantait au boot (F1, corrigé). Conséquence : une part importante
des échecs e2e est du **drift spec↔source** (les specs visent l'ancienne structure d'UI), pas
des régressions produit.

**Cas représentatif :** `cowork-smoke › opens Settings and renders the A2A registry tab`
échoue car il ouvre les réglages via le chemin de menu `Fichier → Paramètres`, alors que le
nouveau `TopMenuBar` a réorganisé l'interaction du dropdown. La feature elle-même est
**intacte** : `data-testid="settings-panel"` existe toujours (`SettingsPanel.tsx:465`) et
s'ouvre via `setShowSettings(true)`. → échec de **test**, pas de produit.

**Résultats (build corrigé, 152 tests dont ~17 gated auto-skippés).** Au point de mesure :
≈ **35 pass / 33 fail** sur la couche smoke/depth/permissions (tests 1–85), puis la couche
**bundles** (tests-runner-*). Le sweep est long (certains bundles ont 1000+ sous-tests) ;
chiffres à lire **avec la réserve « working tree mid-refactor »**.

**Triage — 2 buckets nets :**

| Bucket | Verdict | Exemples |
|---|---|---|
| **Logique in-process** (bundles déterministes) | ✅ **PASSE** | `test-runner-a2a-acp-channel-bundle` → 82 ok ; permissions deny/allow/bash rules (cowork-smoke) → pass |
| **Navigation GUI** (chemins de menu) | ❌ **échec = drift refactor `TopMenuBar`** | `Fichier→Paramètres`, `Outils→Test Runner`, Word-workshop, langue FR, A2A tab |

**Confirmé via l'outil livré :** `cowork-pilot run-bundle code-buddy-cowork-functional-coverage-bundle`
→ `1 ok / 7 ko`. Les 7 KO sont des sous-tests qui pilotent la GUI par les menus refactorés ;
le pilote rapporte fidèlement l'état réel (et a dû lui-même basculer sur les setters store +
`setWorkdir` pour contourner le drift de menu). **La couverture fonctionnelle in-process reste
verte ; ce qui casse, c'est l'UI de navigation en cours de refactor.**

*Base propre recommandée : relancer après stabilisation du refactor — trivial via `cowork-pilot` (§11).*

## 9. Findings & triage

### F1 — CRITIQUE (corrigé) : le working tree crashait Cowork au boot

En reconstruisant depuis la source **non commitée** (43 fichiers `cowork/src` modifiés),
le renderer plantait immédiatement sur une **error boundary** « Something went wrong ».

- **Cause racine :** `ReferenceError: TeamWrapper is not defined` (renderer).
  Un refactor incomplet dans `cowork/src/renderer/App.tsx` avait remplacé `<TeamPanel />`
  par `<TeamWrapper />` (ligne 636) **sans jamais définir** `TeamWrapper` (alors que le
  pattern voisin `FleetCommandCenterWrapper` est défini).
- **Diagnostic :** capturé via `e2e/_audit-boot-diag.spec.ts` (écoute `pageerror`/console).
- **Correctif final :** `TeamWrapper()` **gaté sur `showTeamPanel`**
  (`const open = useAppStore(s=>s.showTeamPanel); if(!open) return null; return <TeamPanel/>;`).
  Un premier jet « minimal » (rendu inconditionnel de `<TeamPanel/>`) faisait booter l'app mais
  a révélé **F4** (voir ci-dessous) → version réactive retenue. Après rebuild :
  `PAGE_ERRORS: (none captured)`, l'app boote, chat réel OK (`CBAUDIT 391`).

> ⚠️ Sans ce correctif, **aucune** fonctionnalité GUI n'est testable sur le working tree actuel.
> Le build de la dernière source cohérente (≈HEAD) fonctionnait. Tous les tests ci-dessous
> ont été (re)lancés contre le build **corrigé**.

### F2 — Sweep e2e contre build périmé (résolu, méthodologie)

Premier sweep lancé contre un `dist-electron` daté de **20:32** alors qu'`App.tsx` avait été
modifié à **20:34** → drift build/spec, ~10 échecs/17 en début de run. Invalidé puis relancé
après rebuild cohérent (voir §6).

### F4 — CRITIQUE (corrigé) : effondrement du layout → contenu principal invisible

**La cause racine de la majorité des échecs e2e.** `TeamPanel` rend un `<div class="flex h-full
w-full">` ; rendu **inconditionnellement** comme enfant direct de `app-root` (flex-col, 900 px), il
occupait **820 px** et écrasait la colonne de contenu principale (`<main>` + DockWorkspace = chat,
settings) à **hauteur 0** → chat ET réglages présents dans le DOM mais **invisibles**
(`toBeVisible()` échoue ; `mainH = 0`).

- **Diagnostic :** sonde `cowork-pilot` de la chaîne d'ancêtres → `app-root` 900 px, enfant `flex-1`
  à 0 px, sibling `team-panel` à 820 px.
- **Correctif :** `TeamWrapper` gaté sur `showTeamPanel` (F1) → par défaut TeamPanel ne monte pas →
  `mainH` repasse à **820 px**, settings-panel **820 px (visible)**, transcript chat visible.
- **Impact mesuré :** `cowork-smoke › opens Settings…` **✘ → ✓** après correctif.

### F5 — CRITIQUE (corrigé) : `SettingsPanel is not defined` (réglages silencieusement cassés)

`App.tsx` rendait `<SettingsPanel>` (l.434, sous `<Suspense>`) **sans l'importer** → `ReferenceError`
**avalé par le `PanelErrorBoundary fallback={null}`** : cliquer *Fichier → Paramètres* mettait
`showSettings=true` mais n'affichait **rien** (échec silencieux, aucune trace visible).

- **Correctif :** ajout de l'import lazy `const SettingsPanel = lazy(() => import('./components/SettingsPanel')…)`.
- Même classe de bug que F1 (composant utilisé non importé, refactor incomplet).

### F6 — typecheck rouge : 6 erreurs `TS6133` (code mort du refactor) corrigées

`tsc --noEmit` (cowork a `noUnusedLocals`) échouait sur 6 déclarations mortes laissées par le
refactor (déplacements de panneaux App↔DockWorkspace) : import `SettingsPanel` + `showSettings` +
`setShowSettings` dans `DockWorkspace`, et `showReasoningViewer` + wrappers `AutonomyWrapper`/
`ReasoningViewerWrapper` (+ imports lazy associés) dans `App.tsx`. Code mort **vérifié** (fonctionnalité
rendue ailleurs) → supprimé. **`tsc --noEmit` : 0 erreur.**

### F3 — (révisé) le menu de navigation fonctionne ; pas de « drift menu »

Diagnostic initial corrigé : la sonde `cowork-pilot` montre que la navigation marche
(*Fichier* → dropdown → *Paramètres* → `showSettings=true`). Les échecs imputés au « refactor menu »
étaient en réalité **F4 + F5**. `TopMenuBar.tsx` (nouveau, non-suivi) et `MemoryBrowser.tsx`
(supprimé) confirment un refactor de navigation en cours, mais les entrées de menu sont fonctionnelles.
Un résidu de drift spec↔source peut subsister sur quelques specs ; à réaligner, trivial via `cowork-pilot`.

## 11. Outillage livré — `cowork-pilot` (CLI + MCP)

Pour rendre les futurs tests de Cowork faciles et reproductibles, un outil de
pilotage de la GUI a été créé sous `cowork/pilot/` (sur Playwright `_electron`,
déjà dépendance Cowork). Il lance sa propre instance isolée **ou** s'attache à
une instance lancée avec `--remote-debugging-port`.

| Fichier | Rôle |
|---|---|
| `cowork/pilot/pilot-core.mjs` | `CoworkPilot` — launch/attach, chat (réel), screenshot, eval, IPC, état store, bundles Test Runner |
| `cowork/pilot/cli.mjs` | CLI `cowork-pilot` — daemon long-vécu + sous-commandes client + mode `once` |
| `cowork/pilot/mcp-server.mjs` | Serveur **MCP** stdio — 12 outils (`cowork_launch/chat/screenshot/eval/ipc/get_state/click/fill/list_bundles/run_bundle/get_config/close`) |
| `cowork/pilot/README.md` | Doc d'usage |

**Validation en conditions réelles (les 3 couches) :**

| Couche | Test | Preuve |
|---|---|---|
| Cœur (`pilot-core`) | `pilot/_smoke.mjs` | chat ChatGPT live → `CBPILOT 391 [tokens: 35 in / 3 out | cost: $0.0000]` ; screenshot `/tmp/cowork-pilot-smoke.png` |
| MCP (`mcp-server`) | `pilot/_mcp_e2e.mjs` (client SDK → stdio) | 12 tools listés ; `cowork_launch(real)` + `cowork_chat` → `CBMCP 391` ; `cowork_screenshot` → image renvoyée (`/tmp/cowork-mcp-e2e.png`) |
| CLI (`cli.mjs`) | self-test daemon + client + `once` | `--help`, daemon `/health` < 2 s, `state`, `shot` (PNG 1400×900), `stop`, `once shot` — tous PASS |

- **Enregistré** dans `~/.codebuddy/mcp.json` (serveur `cowork-pilot`) → utilisable depuis Code Buddy / tout client MCP.
- Le calcul `17×23=391` (jeton imposé `CBxxx`, exclusion de l'écho du prompt) prouve une **vraie sortie modèle**, pas un mock.
- Exemple CLI : `node cli.mjs daemon --real` puis `node cli.mjs chat "17*23 ?" --marker '^CB'`.
